# Explain Translations Implementation Plan

## Overview

Add an on-demand "Explain" feature to the Train and Learn screens: after the
answer is checked (Train) or revealed (Learn), the user can open a modal that
explains why the Spanish phrase maps to the card's stored English translation.
Explanations are generated by **GPT-5.4 mini** via the OpenAI Responses API,
returned as markdown, cached in Postgres keyed on the (Spanish, English) text
pair, and gated by a new `en<->es` language-pair tag on cards. The LLM call is
an injected boundary so unit tests use fakes and e2e uses a local stub server.

Idea document: `meta/ideas/explain-translations.md`.

## Current State Analysis

- **No language-pair concept exists.** `cards` has only `spanish_text` /
  `english_text` (`server/migrations/1718000000000_create-cards.cjs:4-16`).
- **Server is DI-friendly.** `createApp(config, pool)` mounts router factories
  (`server/src/app.ts:29-32`); MCP already injects domain functions as plain
  closures for testability (`McpDeps`, `server/src/mcp/tools.ts:12-15`). The
  LLM client follows the same pattern.
- **Config has an optional-var precedent**: `mcpToken: env.MCP_TOKEN?.trim()
  || null` with a startup warning instead of a crash (`server/src/config.ts:31`,
  `server/src/index.ts:34-36`). `OPENAI_SECRET_KEY` follows it. The key already
  exists in the local `.env`.
- **Train reveal state**: `reveal !== null` on `TrainPage`
  (`client/src/training/TrainPage.tsx:36,82-86`); the post-check block renders
  `<AnswerReveal>` + `<RatingBar>` (`TrainPage.tsx:190-198`).
- **Learn reveal state**: `showBack` on `LearnPage`
  (`client/src/learning/LearnPage.tsx:43,174-184`); the answer slot keeps
  `visibility: hidden` when concealed so layout never shifts (matches the
  stable-layout preference and the height assertion in
  `e2e/learning.spec.ts`).
- **No modal exists anywhere in the client**; this introduces the first one.
  Window-level keydown handlers that must be suppressed while it is open:
  Space/1/2 on `LearnPage.tsx:102-124` and 0–3 in
  `client/src/training/RatingBar.tsx:42-60` (already gated by its `disabled`
  prop).
- **Client API layer** is one fetch wrapper + exported functions and DTO
  interfaces in `client/src/api.ts`; pages use `LoadState` strings and an
  `ApiError`-based 401 handler.
- **Tests**: server/client unit tests are Vitest with DI fakes, never a real
  DB. E2E (Playwright, `workers: 1`) starts the real API + Vite via the
  `webServer` array (`playwright.config.ts:28-48`) against an isolated compose
  Postgres; specs seed via the real HTTP API. There is no HTTP-stubbing
  precedent — the OpenAI stub server is new.

## Desired End State

- During Train, after checking an answer, an "Explain «spanish text»" button
  appears for `en<->es` cards; during Learn it occupies a stable slot and
  becomes visible once the answer is shown.
- Clicking it (or pressing `E`) opens a modal: spinner while generating, then
  a markdown-rendered explanation; on any failure the modal shows exactly
  **"Sorry! Something went wrong with this explanation."**
- The first request for a given (Spanish, English) pair calls GPT-5.4 mini and
  saves the result; every later request for the same pair (any card) is served
  from Postgres with no model call.
- `pnpm test`, `pnpm typecheck`, and `pnpm e2e` all pass with **zero** real
  LLM calls.

### Key Discoveries

- `gpt-5.4-mini` (current cheap tier, $0.75/$4.50 per 1M tokens in/out) ≈
  $0.0016 per explanation; call via `client.responses.create` with
  `reasoning: { effort: 'none' }` pinned (openai npm v6, SDK supports
  `baseURL` override — that's the e2e stub hook).
- `TrainingCard` is a separate slimmer DTO from `Card` on both sides
  (`server/src/training/repository.ts:7-12`, `client/src/api.ts:32-37`) —
  `languagePair` must be added to both queries/types.
- Explanations need a card lookup by id (texts + language pair); no
  single-card getter exists in `server/src/cards/repository.ts` yet.
- A shared cache table (no FK to `cards`) survives the app's delete-and-
  recreate card workflow and deduplicates identical pairs across cards — and
  it is no more complex than per-card storage, so we take it (the idea doc
  delegated this choice to the planner).

## What We're NOT Doing

(Per the idea doc's non-goals.)

- No explanations on card creation/editing screens, no chat follow-ups, no
  regeneration/refresh controls, no model-selection UI, no daily limits.
- No pre-generation in the background; generation happens only on click.
- No user-editable explanations; content is read-only.
- No explanation invalidation (cards are deleted, not edited).
- No language pairs beyond `en<->es`; the tag exists so future pairs can opt
  out, nothing more.
- No rich/custom markdown pipeline — `react-markdown` with defaults (it does
  not render raw HTML, so stored content is safe to display). If it proves
  awkward during implementation, fall back to rendering the raw markdown text
  in a `white-space: pre-line` block and defer rendering to a follow-up task
  (explicitly acceptable per user direction).

## Implementation Approach

Four phases, each independently verifiable: (1) the `language_pair` tag end to
end, (2) the server-side explanation API with the injected LLM boundary,
(3) the client button + modal, (4) the e2e OpenAI stub and specs plus docs/env
sync. DB work follows the existing node-pg-migrate `.cjs` pattern with real
`down` migrations.

#### 4. LLM boundary

**File**: `server/src/explanations/llm.ts` (new)

```ts
import OpenAI from 'openai';
import type { AppConfig } from '../config.js';

export const EXPLANATION_MODEL = 'gpt-5.4-mini';

// Generates explanation markdown for a Spanish phrase / English translation
// pair. Null when OPENAI_SECRET_KEY is not configured; throws on API failure.
export type ExplanationGenerator = (
  spanishText: string,
  englishText: string,
) => Promise<string>;

const INSTRUCTIONS = [
  'You are a concise Spanish grammar tutor helping an English speaker memorize flashcards.',
  'You are given a Spanish word or phrase and the English translation the learner memorizes for it.',
  'Explain why the Spanish supports that English translation: break the phrase into meaningful chunks',
  'and add brief grammar notes (reflexives, articles, tense, idiom, word order) only where they help.',
  'For a single vocabulary word, give a short note on usage, gender, or memorable structure instead of a breakdown.',
  'Treat the provided English translation as the answer being explained; do not propose a different translation as the main output.',
  'Respond in GitHub-flavored markdown using short bullet points. Be scannable and brief: usually 3-6 bullets, no headings, no preamble.',
].join(' ');

export function createExplanationGenerator(config: AppConfig): ExplanationGenerator | null {
  if (!config.openaiSecretKey) {
    return null;
  }
  const client = new OpenAI({
    apiKey: config.openaiSecretKey,
    ...(config.openaiBaseUrl ? { baseURL: config.openaiBaseUrl } : {}),
    timeout: 20_000,
    maxRetries: 1,
  });
  return async (spanishText, englishText) => {
    const response = await client.responses.create({
      model: EXPLANATION_MODEL,
      instructions: INSTRUCTIONS,
      input: `Spanish: ${spanishText}\nEnglish translation to explain: ${englishText}`,
      max_output_tokens: 600,
      reasoning: { effort: 'none' },
    });
    const text = response.output_text?.trim();
    if (!text) {
      throw new Error('Empty explanation from model');
    }
    return text;
  };
}
```

Notes: instructions are a stable prefix (OpenAI prompt caching discounts it);
`reasoning.effort` is pinned to `'none'` so a future SDK/model default change
cannot silently add latency/cost; `maxRetries: 1` keeps worst-case modal wait
bounded.

#### 5. Repository

**File**: `server/src/explanations/repository.ts` (new)
- `findExplanation(db, spanishText, englishText): Promise<Explanation | null>`
  — SELECT by the unique pair.
- `insertExplanation(db, { spanishText, englishText, contentMarkdown, model })`
  — `INSERT ... ON CONFLICT (spanish_text, english_text) DO NOTHING
  RETURNING ...`, and on conflict (concurrent double-click race) fall back to
  `findExplanation` so the first writer wins and both callers return the same
  row.
- `Explanation` domain type: `{ id, spanishText, englishText,
  contentMarkdown, model, createdAt }`, snake→camel mapper like `toCard`.

#### 6. Service

**File**: `server/src/explanations/service.ts` (new)

```ts
export interface ExplanationDeps {
  findExplanation: (spanish: string, english: string) => Promise<Explanation | null>;
  insertExplanation: (input: NewExplanation) => Promise<Explanation>;
  generate: ExplanationGenerator | null;
}

export type ExplanationResult =
  | { status: 'ok'; explanation: Explanation; source: 'cached' | 'generated' }
  | { status: 'unavailable' }; // generator unconfigured

export async function getOrCreateExplanation(
  deps: ExplanationDeps,
  spanishText: string,
  englishText: string,
): Promise<ExplanationResult>;
```

Logic: cached row → return `cached` (never calls `generate`); no generator →
`unavailable`; otherwise generate (errors propagate to the route), insert,
return `generated`.

#### 7. Card lookup + routes

**File**: `server/src/cards/repository.ts`
- Add `getCard(db, id): Promise<Card | null>` (same SELECT shape as
  `listCards` with `WHERE c.id = $1`).

**File**: `server/src/explanations/routes.ts` (new)
- `explanationRoutes(pool, generator)` returning a Router with
  `POST /:id/explanation`:
  - non-integer id → 400; card not found → 404 `{ error: 'Card not found' }`
  - `card.languagePair !== 'en<->es'` → 400
    `{ error: 'Explanations are not supported for this card type' }`
  - `unavailable` → 502 `{ error: 'Explanation generation is not configured' }`
  - generator throw → 502 `{ error: 'Explanation generation failed' }`
    (caught in the route; log the underlying error, never expose it)
  - success → 200 `{ explanation: { contentMarkdown, model, createdAt },
    source: 'cached' | 'generated' }`

POST (not GET) because a cache miss creates a row. The client maps **any**
failure to the agreed friendly message, so status nuance is for logs/tests.

**File**: `server/src/app.ts`
- Build the generator once and mount alongside card routes:

```ts
app.use('/api/cards', requireAuth(config), cardRoutes(pool));
app.use('/api/cards', requireAuth(config), explanationRoutes(pool, createExplanationGenerator(config)));
```

#### 8. Unit tests

**File**: `server/tests/explanations/service.test.ts` (new)
- cached pair → returns `cached`, generator never invoked (spy)
- cache miss → generates, inserts, returns `generated`
- `generate: null` → `unavailable`
- generator rejection propagates

**File**: `server/tests/explanations/routes.test.ts` (new)
- mount `explanationRoutes` on a throwaway express app with fake deps (the
  `server/tests/mcp/mcp.test.ts` pattern), covering 404, wrong-language 400,
  502 on failure/unconfigured, and the 200 shape. Requires the route file to
  accept its repository/lookup functions injectably — implement
  `explanationRoutes(pool, generator, overrides?)` or factor the handler so
  deps are constructed per-request from the pool but replaceable in tests
  (follow whichever the MCP precedent makes cleanest).

## Phase 4: E2E OpenAI Stub, Specs, Env/Docs Sync

### Overview

A local fake of the OpenAI Responses endpoint wired into the e2e stack via
`OPENAI_BASE_URL`, new e2e coverage, and the mandatory env-example/docs

### Changes Required

#### 1. Stub server

**File**: `e2e/openai-stub.ts` (new) — a tiny node `http` server:
- `POST /responses` → if the request body's `input` contains the sentinel
  string `TRIGGER-EXPLAIN-FAILURE`, respond 500; otherwise return a minimal
  valid Responses-API payload whose aggregated output text is a fixed
  markdown explanation (shape:
  `{ id, object: 'response', status: 'completed', output: [{ type: 'message',
  role: 'assistant', content: [{ type: 'output_text', text: '- **stubbed**
  explanation for e2e' }] }], ... }` — verify the exact minimum the openai v6
  SDK needs to populate `output_text` during implementation).
- `GET /__health` → 200 (webServer readiness URL)
- `GET /__requests` → `{ count }` of `/responses` calls since last reset
- `POST /__reset` → zero the counter
- Port from `process.env.OPENAI_STUB_PORT`.

#### 2. E2E env plumbing

**File**: `.test-env.example` (and local `.test-env`)
- Add `OPENAI_STUB_PORT=4115` with a comment. **Run `freeport 4115` before
  committing to it** (host rule); if taken, walk up and update the port
  table in `CLAUDE.md` accordingly.

**File**: `e2e/env.ts`
- Export `E2E_OPENAI_STUB_PORT = Number(env('OPENAI_STUB_PORT', '4115'))`.

**File**: `playwright.config.ts`
- Third `webServer` entry: `command: 'npx tsx e2e/openai-stub.ts'`, `url:
  http://localhost:${E2E_OPENAI_STUB_PORT}/__health`, env

**File**: `e2e/explain.spec.ts` (new) — same `logIn`/`wipeAllCards`/
`createCard` helpers as `e2e/training.spec.ts`; `POST /__reset` on the stub in

- **Train gating + happy path + cache**: create card; in `/train` assert no

- **Learn gating**: in `/learn`, explain button hidden/inert before

- **Failure path**: card whose Spanish text contains

## Performance Considerations

- One model call per *novel* text pair, ever; everything else is a single
  indexed SELECT. At ~$0.0016/call (GPT-5.4 mini, ~300 in/out tokens) cost is
  negligible for a single-user app.
- 20s SDK timeout + 1 retry bounds the modal's worst-case loading state;
  stable instruction prefix earns OpenAI's cached-input discount on the
  prompt's fixed part.

## Migration Notes

- Both migrations are additive with real `down`s; `pnpm migrate:up` on dev,
  and prod runs migrations automatically at container start
  (`docker-compose.yml` command), so `pnpm ship` picks them up.
- Prod requires one manual env addition (`OPENAI_SECRET_KEY` in
  `.prod.app.env` on the prod checkout) **before** the feature works there;
  the app degrades gracefully (warning + friendly error) if shipped first.
